## 4. 남은 사람 작업

### A. 데모 영상

1. 실제 운영 제품을 430 × 932 모바일 화면으로 녹화
2. 2분 45초~2분 55초 편집
3. 영어 내레이션 또는 검수된 영어 자막 추가
4. 실제 GPT-5.6 평가 출처 라벨 포함
5. Codex 협업과 GitHub 기록 포함
6. YouTube 공개 또는 일부 공개 업로드
7. 로그아웃 상태에서 재생 확인

실행 정본: `docs/submission/DEMO_VIDEO_HANDOFF_KO.md`

### B. Codex Session ID

1. 핵심 제품 개발이 이뤄진 Codex 작업에서 `/feedback` 실행
2. 반환된 정확한 Session ID 복사
3. `DEVPOST_COPY.md`와 Devpost 폼에 동일하게 입력

폴더명, 첨부 ID, 브라우저 세션 ID를 추정해서 넣지 않는다.

### C. 저장소 라이선스

현재 라이선스가 없다. 소유자가 선택해야 한다.

- 추천 기본안: MIT
- 선택 후 저장소 루트에 `LICENSE` 추가
- 공개하고 싶지 않은 교육 콘텐츠·데이터의 권리 범위가 있는지 먼저 확인

Codex가 소유자 승인 없이 라이선스를 임의 추가하지 않는다.

### D. Devpost 최종 제출

1. 영문 문안 붙여넣기
2. Education 트랙 확인
3. 운영 URL, GitHub, YouTube URL 확인
4. 정확한 Session ID 입력
5. 소유권·동의 항목 확인
6. 제출 전 최종 화면 캡처
7. 소유자가 Submit

## 8. 최종 제출 전 리허설

새 시크릿 브라우저에서:

- [ ] 운영 URL이 열린다.
- [ ] 모바일 가로 스크롤이 없다.
- [ ] 입력 글자 크기가 iPhone 자동 확대를 만들지 않는다.
- [ ] 듣기 음성이 재생된다.
- [ ] 진단 결과가 저장된다.
- [ ] Google·이메일 로그인 경로가 현재 안내와 일치한다.
- [ ] 실제 GPT-5.6 평가가 한 번 성공한다.
- [ ] GitHub 저장소가 공개다.
- [ ] YouTube 영상이 로그인 없이 재생된다.
- [ ] 영상 길이가 3분 미만이다.
- [ ] Devpost의 모든 URL이 새 탭에서 열린다.
- [ ] Session ID가 정확하다.
- [ ] 라이선스 선택이 반영됐다.

## 9. 해커톤 직전 코드 변경 원칙

- 새 기능을 추가하지 않는다.
- 심사를 막는 오류만 작은 브랜치에서 수정한다.
- 모바일·인증·AI 평가·데이터 유지 문제를 우선한다.
- 수정 시 `npm run lint`, `npm test`, 두 모바일 뷰포트 검증 후 PR로 머지한다.
- 제출 문안과 실제 화면이 달라지면 문안도 함께 갱신한다.

## 10. 해커톤 이후 릴리스 순서

1. 첫 코호트 10가정 시작
2. 운영 오류와 이탈 수집
3. A 코호트 치명적 문제만 수정
4. B·C 코호트 확대
5. 3일 반복과 WPA 확인
6. 부모 인터뷰 5건
7. 결제·법률·보호자 확인 방식 검토
8. 검수된 콘텐츠 범위 확장

결제는 코드가 쉽다는 이유로 먼저 열지 않는다. 운영자 정보, 환불정책, 보호자 확인, PG 계약을 확인한 뒤 별도 릴리스로 진행한다.
