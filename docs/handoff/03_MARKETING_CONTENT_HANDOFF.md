# 15Loop 개발기 기반 마케팅 핸드오프

*목적: 15Loop의 실제 개발 기록을 부모 모집과 Project JYP 리드 생성에 재사용하는 콘텐츠 운영 정본*

## 1. 마케팅의 중심 자산

광고 문구를 새로 지어내는 것이 아니라 다음 사실을 반복해서 다른 관점으로 보여준다.

> 선행을 시키지 않은 아빠가 중학교에 간 아이의 영어 문제를 발견했고, AI와 함께 제품을 만들었지만 실제 아이가 써보자 가설이 틀렸다는 것을 알았다. 그래서 기능을 줄이고, 가입 순서를 바꾸고, 평가 결과가 다음 학습을 바꾸게 만들었다.

이 개발기는 두 시장에서 다른 역할을 한다.

- 부모에게는 “이 사람도 같은 문제를 실제로 겪었다”는 공감과 무료 진단의 이유가 된다.
- 바이브 코딩 관심자에게는 “AI로 화면만 만든 것이 아니라 사용자·데이터·가입·검증까지 출시했다”는 Project JYP의 증거가 된다.

## 2. 절대 섞지 않을 두 콘텐츠 트랙

### Track P — 부모·15Loop

- 대상: 초5·6 또는 중1 자녀의 단어 학습이 걱정되는 부모
- 문제: 뜻은 외웠는데 읽기·듣기·문장에서 막힘
- 약속: 가입 없이 어디에서 연결이 끊기는지 확인
- CTA: `우리 아이 무료 진단하기`
- 성공지표: 진단 완료, 부모 연결, 첫 학습, 3일 완료, WPA

### Track B — 빌더·Project JYP

- 대상: 바이브 코딩은 해봤지만 서비스 출시와 고객 검증에서 막힌 사람
- 문제: 코드는 만들었지만 누구에게 무엇을 팔지, 가입·데이터·배포·검증을 어떻게 할지 모름
- 약속: 실제 15Loop 의사결정과 수정 과정을 실행 프레임으로 제공
- CTA: 자기 프로젝트 문제와 사용자를 한 문장으로 제출
- 성공지표: 유효 댓글·DM, 이메일 신청, 프로젝트 브리프, 첫 빌드 행동, WAB

한 게시물에는 한 대상과 한 CTA만 둔다. 같은 개발 장면을 두 버전으로 재작성할 수는 있지만 본문에서 두 서비스를 동시에 판매하지 않는다.

## 3. 채널 역할

### Threads — 이야기와 대화

- 주 채널: `@jyp_lab`
- 개발 장면, 실패, 당일 숫자, 부모 질문을 짧게 공개
- 링크는 본문보다 첫 댓글을 우선
- 자동 발행 금지, 계정 소유자가 최종 확인 후 발행

### 15loop.com — 부모 전환

- 부모 콘텐츠의 유일한 최종 목적지
- 무료 진단이 첫 화면
- UTM을 붙여 게시물별 진단·부모 연결 기여를 측정

### GitHub — 빌드 증거

- 빌더 대상 게시물에서 커밋·테스트·PR 근거로 사용
- 부모 게시물에서는 기술 자랑을 앞세우지 않는다.

### 향후 블로그·검색 — 축적

- 개발기 장문, 부모 FAQ, 단어 연결 교육 콘텐츠를 검색 가능한 형태로 축적
- 현재 즉시 실행 우선순위는 Threads와 운영 랜딩이다.

## 4. 콘텐츠 기둥

### Pillar 1 — 외웠는데 모르는 이유

**핵심 질문:** 왜 뜻은 아는데 읽거나 들으면 모를까?

**연결되는 제품:** 네 연결 진단

**주제 묶음:**

- 종이 단어장이 뜻과 소리를 분리하는 순간
- recognition과 listening이 다른 이유
- 문장에서 알던 단어가 사라지는 이유
- 떠올리기와 객관식 정답의 차이
- 부모가 집에서 관찰할 수 있는 세 가지 장면

**성격:** 검색 가능 + 공유 가능

### Pillar 2 — 선행 안 시킨 아빠의 개발기

**핵심 질문:** 왜 아빠가 직접 만들게 됐나?

**연결되는 제품:** 창업자 신뢰와 무료 진단

**주제 묶음:**

- 중1 아이에게 단어를 읽어보라고 한 날
- 하루 30단어 목표를 버린 이유
- 아이 계정이 아니라 부모 계정으로 바꾼 이유
- LoopVoca에서 15Loop로 이름을 바꾼 날
- 5일 동안 오픈베타까지 만든 기록

**성격:** 공유 가능, 메타 콘텐츠

### Pillar 3 — 아이가 가설을 깨뜨린 순간

**핵심 질문:** 실제 사용자가 제품을 어떻게 바꿨나?

**연결되는 제품:** cloze recall, 선택 AI 도전, 모바일 흐름

**주제 묶음:**

- “이 네 단계만 하는 앱인 줄 알았다”는 반응
- 문장 전체 쓰기를 없앤 이유
- 작은 글씨가 맞는 디자인이 아닌 이유
- 무료 진단을 가입보다 앞으로 옮긴 이유
- 기능을 더하는 것보다 줄이는 것이 어려웠던 장면

**성격:** 공유 가능, 실패·수정 기록

### Pillar 4 — AI 시대의 영어 평가는 무엇인가

**핵심 질문:** AI가 영어를 가르치는 대신 무엇을 평가해야 하나?

**연결되는 제품:** GPT-5.6 제한 평가, 적응형 큐, 로컬 대체 평가

**주제 묶음:**

- AI 생성보다 AI 평가를 택한 이유
- 문자열이 달라도 뜻이 맞는 답
- AI가 결정하지 못하게 남겨둔 것
- 평가 출처를 화면에 표시한 이유
- 학습 데이터가 다음 복습을 바꾸는 구조

**성격:** 검색 가능 + 공유 가능, thought leadership

### Pillar 5 — 오픈베타를 정직하게 운영하는 법

**핵심 질문:** 조회수와 무료 가입 말고 무엇을 봐야 하나?

**연결되는 제품:** 7일 프로그램, 완료 학습일, WPA

**주제 묶음:**

- 한 번의 무료 진단으로 결제받지 않는 이유
- 가격을 체험 전에 공개한 이유
- streak 대신 실제 완료 학습일을 고른 이유
- 3,000단어를 모으고도 30개만 공개한 이유
- 베타 미달 숫자도 공개해야 하는 이유

**성격:** 공유 가능, 빌더 리드 생성

## 6. 시즌 1 권장 구성

### 1막 — 문제를 발견하다

1. 선행을 시키지 않은 아빠의 불안
2. 단어장에는 뜻만 있고 소리는 다시 찾아야 했던 문제
3. “모른다”가 아니라 “연결되지 않았다”는 가설

### 2막 — 빠르게 만들다

4. 첫날 LoopVoca MVP
5. 하루 30단어에서 15분으로
6. 아이 가입에서 부모 계정으로
7. LoopVoca에서 15Loop로

### 3막 — 실제 아이가 부수다

8. 네 단계 반복이 제품 전체로 보였던 문제
9. 문장 쓰기를 cloze와 선택 도전으로 줄임
10. 무료 진단을 가입 앞으로 이동

### 4막 — 서비스가 되다

11. 평가가 다음 큐를 바꾸게 만든 과정
12. 3,000개를 모으고 30개만 공개한 이유
13. 보호자·개인정보·AI 출처 설계
14. 검색 등록, 분석, 모바일, 배포

### 5막 — 시장에 묻다

15. 7일 베타와 12,900원 가설
16. 첫 코호트 숫자 공개
17. 부모 인터뷰와 아이의 재사용 이유
18. 유지할 것과 버릴 것

## 7. 발행 리듬

기본 운영:

- 주 3개 핵심 게시물: 부모 2개, 빌더 1개
- 각 핵심 게시물에서 후속 댓글·짧은 업데이트 1~2개
- 주 1회 실제 숫자와 배운 점 정리

`베타 1,000명` 공개 도전을 별도로 실행할 경우:

- 매일 숫자 업데이트는 짧은 운영 로그로 취급한다.
- 핵심 개발기 게시물은 주 3개 원칙을 유지한다.
- 하루에 부모·빌더 두 CTA를 연속 하드셀하지 않는다.
- 1,000명 달성 여부와 제품 검증 여부를 분리해 공개한다.

## 9. 콘텐츠 재가공

개발기 한 에피소드에서 다음 자산을 만든다.

- Threads 원문 1개
- 부모 버전 또는 빌더 버전 1개
- 30~45초 세로 영상 대본 1개
- 블로그 장문에 추가할 단락 1개
- FAQ 또는 랜딩 반론 문장 1개
- 향후 이메일 한 문장 1개

같은 내용을 복사하지 않고 대상의 문제와 CTA만 바꾼다.

## 10. UTM과 측정

### 부모 트랙

```text
utm_source=threads
utm_medium=organic_social
utm_campaign=15loop_open_beta_2026q3
utm_content={episode_id}_parent
```

게시물별로 진단 시작보다 진단 완료, 부모 연결, 3일 완료를 우선 평가한다.

### 빌더 트랙

```text
utm_source=threads
utm_medium=organic_social
utm_campaign=vibe_build_2026q3
utm_content={episode_id}_builder
```

조회수보다 유효 댓글·DM, 이메일 신청, 브리프 제출, 첫 행동을 본다.

## 11. 숫자 공개 원칙

- 추정과 실측을 구분한다.
- 팔로워·조회·가입·진단·부모 연결을 같은 숫자로 부르지 않는다.
- “베타 가입”은 부모 계정 생성인지 진단 완료인지 게시물에 정의한다.
- 미달도 그대로 공개한다.
- 학원비, 주변 부모 반응, 자녀의 말은 실제 확인한 경우만 인용한다.
- 아이의 점수와 이름을 공개하지 않는다.

## 14. 완료 정의

- 18편 시즌 구조가 부모·빌더 트랙으로 분리돼 있다.
- 첫 10편의 증거와 CTA가 지정돼 있다.
